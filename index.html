mcp-test-runner github demo readme

$ mcp-test-runner --serve

一個 Model Context Protocol server。讓任何 MCP client（Claude Desktop / Cursor / Codex CLI / Gemini CLI / Cline / Zed ...）直接驅動 pytest / Jest / Cypress / Go test。每跑完一次就告訴你下一步該改什麼。

寫這個的動機很單純：我不想再重複「跑測試 → 看 log → 整理失敗 → 想下次該補哪些」 這個迴圈。MCP 讓 Claude 接手執行 + 鑑識 + 教練三件事；我只要回答它指過來的「下一步」。

what it does

三件事，按你會用到的頻率排：

1. 跑測試

透過 QA_RUNNER 環境變數切 framework。pytest-playwright 已完整實作（含 auto-retry / tracing / JUnit XML）；jest / cypress / go test 用同一介面、各自有 runner plugin 處理原生輸出。每條 test 都拿 screenshot + video + Playwright trace.zip。

2. 從 URL 寫測試

analyze_url 用 headless Chromium 打開頁面、拆 form / nav / dialog / CTA 模塊，順便監聽 fetch/xhr 收集 API endpoints。每個模塊附上候選 TC 清單。 generate_test 接過去寫 runnable pytest 骨架 — 用真實的 selectors， 不是 # TODO 由 AI 補完 占位符。

3. 當你的教練

每跑完一次 _archive_report() 自動把 report.json 快照進 test-results/history/，並寫一份新的 optimization-plan.md。 Optimizer 從三個視角看你的工作：哪些 test 是 flaky / broken / slow_regression、 哪些 MCP tool 呼叫模式可以包成 meta-tool、AI 產出的測試最後有沒有被採用。

the pipeline

   [URL]
        │  analyze_url
        ▼
   modules + candidate TCs + API endpoints
        │  generate_test (module=...)
        ▼
   tests/test_*.py   ← 可直接跑、不是 # TODO 占位
        │  run_tests
        ▼
   report.json + screenshots + trace.zip + junit.xml
        │
        ├──→ generate_html_report  ──→  self-contained HTML
        ├──→ auto archive          ──→  history/ snapshot
        └──→ optimizer auto-write  ──→  optimization-plan.md
                                                 │
                                                 ▼
                                      下一個 session 讀進來 → loops

迴圈是設計重點。每次 run 餵 optimizer、optimizer 指出最弱環節、下次 run 優先攻那一環。 沒有這個迴圈的話、AI 只是個更快的 monkey tester。

install

git clone https://github.com/kao273183/mcp-test-runner.git
cd mcp-test-runner
python -m venv .venv && source .venv/bin/activate
pip install -e .
playwright install chromium
# 選用：啟用 flaky test auto-retry
pip install pytest-rerunfailures

接到 MCP client

下面是 Claude Desktop 的範例 — 路徑： ~/Library/Application Support/Claude/claude_desktop_config.json （macOS；Windows 在 %APPDATA%\Claude\）。 其他 client 範例：Codex CLI (TOML) · Gemini CLI。 

{
  "mcpServers": {
    "mcp-test-runner": {
      "command": "/path/to/.venv/bin/python",
      "args": ["-m", "mcp_test_runner.server"],
      "cwd": "/path/to/mcp-test-runner",
      "env": {
        "QA_RUNNER": "pytest",
        "QA_PROJECT_ROOT": "/path/to/your-test-project"
      }
    }
  }
}

重啟你的 MCP client（Claude Desktop / Codex / Gemini ...），開 session，試試任一條：

# 一鍵：分析 URL 並產出可執行測試
測試 https://your-site/login — 對每個模塊產 1 條 test

# 看 optimizer 認為下一步該做什麼
下一步該優化什麼？

# 結合領域知識寫回歸測試
參考 qa 知識的歷史 bug 區段，幫我寫 coupon 連點兩次的回歸測試

qa knowledge in three layers

光看 DOM 的 analyzer 只能產「空欄位送出應出錯」這種泛例。這就是 monkey testing 換個包裝。要寫出真 QA 工程師會寫的測試、必須注入領域知識。

所以 MCP 疊了三層：

  • built-in — ISTQB 七大原則、等價分割 / 邊界值、 決策表、狀態轉換、測試金字塔、Shift-Left、回歸策略、Mobile checklist、QA metrics。 內建在 server，沒設定任何東西時 get_qa_context 回的就是這份。
  • your file — 受測專案根的 qa-knowledge.md。 業務規則、歷史 Bug、標準斷言文字、user journey、技術約束。執行 init_qa_knowledge 一鍵起手範本（已含通用方法論 + 5 個 TODO 區段）。
  • per-test inline — 把領域知識 slice 透過 business_context 帶進 generate_test、會自動印成 # Business context: 註解區塊在 test 函式裡。產出的 test 自帶設計依據、 reviewer 不用切換到別的 wiki 才看得到「為什麼測這個」。

三層搭配：built-in 告訴你「怎麼測」、your file 告訴你「測什麼」、 inline 把這兩個答案固化進產出物。任何一層缺失都會退回 monkey testing。

the coach (optimizer)

每次 run 結束、optimizer 讀 history/ + telemetry log、寫一份排序好的行動清單。 三個視角：

  • suite quality — 每條 test 的 outcomes 字串 （PFPFP）算 flake_score、加上 error signature 比對。 連 3 次失敗 + 相同 signature → 標 broken（真 bug、不是 flake）。
  • mcp usability — Tool 呼叫 telemetry 找 top tool、錯誤率、重複 args、常見鏈（A→B 連續呼叫）。提示哪裡可以包 meta-tool 或加 cache。
  • ai effectiveness — generate_test 寫的測試 有沒有出現在下次 run？analyze_url 偵測的模塊有沒有對應 test 檔？ 追蹤採用率 vs 覆蓋缺口。

你可以直接對 Claude 說「下一步該優化什麼」、它會 call get_optimization_plan 然後給你帶 priority + evidence + suggestion 的具體答案。不是憑感覺、是看資料。
