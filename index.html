What's in the box

一個 MCP server、多個介面。用自然語言對 Claude 下指令，不用記 tool 名稱也能跑全套 QA 流程。

🧪

Multi-runner 支援

pytest / Jest / Cypress / Go test — 用 QA_RUNNER 環境變數切換。每個 runner 有對應的 plugin 處理原生輸出格式。

🔍

DOM 分析器

給 URL → 自動拆 form / nav / dialog / CTA 模塊 + 候選 TC 清單 + 偵測到的 API endpoints。從此不用手寫 locator。

🎬

失敗鑑識

每條 test 都有 screenshot + video + Playwright trace.zip。HTML 報告含步驟列表跟 base64 內嵌截圖，packaged 給 reviewer 即用。

📊

自我強化教練

每跑完一次自動寫 optimization-plan.md。三層分析：flake / broken / coverage_gap / slow_regression 排序、告訴你哪條該先修。

End-to-end flow

迴圈是設計重點：每次 run 餵 optimizer、optimizer 指出最弱環節、下一次 run 優先攻那一環。 

   [URL]
        │  analyze_url
        ▼
   modules + candidate TCs + API endpoints
        │  generate_test (module=...)
        ▼
   tests/test_*.py  (可直接執行、不再是 # TODO 占位)
        │  run_tests
        ▼
   report.json + screenshots + trace.zip + junit.xml
        │
        ├──→ generate_html_report  ──→  self-contained HTML report
        ├──→ auto archive          ──→  history/ snapshot
        └──→ optimizer auto-write  ──→  optimization-plan.md
                                                  │
                                                  ▼
                                       next session reads → loops

Quick start

三步從 clone 到「Claude 已經幫你跑完測試」。

1

Install

Clone repo、editable install、裝 Playwright 瀏覽器。

git clone https://github.com/kao273183/mcp-test-runner.git
cd mcp-test-runner
python -m venv .venv && source .venv/bin/activate
pip install -e .
playwright install chromium
# 選用 — 啟用 flaky test 自動重試
pip install pytest-rerunfailures
2

接到 Claude Desktop

把以下加入 ~/Library/Application Support/Claude/claude_desktop_config.json (macOS)。

{
  "mcpServers": {
    "mcp-test-runner": {
      "command": "/path/to/.venv/bin/python",
      "args": ["-m", "mcp_test_runner.server"],
      "cwd": "/path/to/mcp-test-runner",
      "env": {
        "QA_RUNNER": "pytest",
        "QA_PROJECT_ROOT": "/path/to/your-test-project"
      }
    }
  }
}
3

對 Claude 下指令

重啟 Claude Desktop。打開 session，試試下面任一條：

# 一鍵：分析 URL 並產出可執行測試
"測試 https://your-site/login — 對每個模塊產 1 條 test"

# 看 optimizer 認為下一步該做什麼
"下一步該優化什麼？"

# 結合領域知識寫回歸測試
"參考 qa 知識的歷史 bug 區段，幫我寫 coupon 連點兩次的回歸測試"

跳脫 monkey testing 的三層 QA 知識

光看 DOM 只能產「空欄位送出應出錯」這種通用測試。要做真 QA、就得把領域知識疊上來。

Layer 1 — Built-in

通用測試方法論

ISTQB 七大原則、等價分割 / 邊界值、決策表、狀態轉換、測試金字塔、Shift-Left、回歸策略、Mobile checklist、QA metrics。MCP 內建、零設定。

Layer 2 — Your file

領域業務知識

業務規則、歷史 Bug、標準斷言文字、user journey。寫在受測專案的 qa-knowledge.md，用 init_qa_knowledge 一鍵起手。

Layer 3 — Per-test

單條測試 inline 註解

把領域 slice 透過 business_context 帶進 generate_test，會自動印成 # Business context: 註解區塊。test 自帶設計依據。

The self-improvement coach

每次 run 結束自動快照進 history，optimizer 從三個視角產優先級行動清單。

Lens 1

測試套件品質

每條 test 的 outcomes 字串（PFPFP）→ 算 flake_score、分類 broken / flaky / slow_regression / stable_passing。連 3 次失敗 + 相同 error signature → 真 bug，不是 flake。

Lens 2

MCP 使用模式

從 telemetry 看 top tool、錯誤率、重複 args 模式、常見鏈（A→B 連續呼叫）。提示哪裡可以包 meta-tool 或加 cache。

Lens 3

AI 產測效益

generate_test 寫出的測試有沒有進到下次 run？analyze_url 偵測到的 module 有沒有對應 test 檔？追蹤採用率 vs 覆蓋缺口。
