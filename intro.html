v0.1 · 公測 Open Source · MIT Model Context Protocol

把 QA 工作流，接到你的 AI 編輯器

QA MCP Server 是 TestBridge 釋出的開源 Model Context Protocol 伺服器。 讓 Claude、Cursor、Windsurf 等支援 MCP 的 AI 編輯器， 可以直接在對話中執行測試、解析失敗、產生 test case。 內建 Playwright、Jest、Cypress、Go test 四種 runner， 透過環境變數即可切換。

安裝指南 技術架構

為什麼釋出這個

TestBridge 雲端平台主打 AI 案例生成、測試管理、QA 社群。 但實際 QA 工作中，測試案例與測試執行往往分散在多個工具： 編輯器在寫程式碼、terminal 在跑測試、CI 介面在看報告、Slack 在回報結果。

QA MCP Server 把「執行測試」這件事直接內嵌到 AI 編輯器。 工程師不用切換視窗，就能透過自然語言請 AI 跑測試、看失敗、補測試。 它是 TestBridge 生態的本地端工具，未來會與雲端平台整合 （例如將執行結果同步到 TestBridge 儀表板）。

SCOPE定位

QA MCP Server 處理本地測試執行與 AI 互動； TestBridge 雲端平台處理測試案例管理、團隊協作與儀表板。 兩者獨立可用，未來整合後互補。

核心功能

8 個 MCP Tools，4 個內建 Runner，所有 AI 編輯器以同一組介面操作。

TOOLS
8 個 MCP 工具
列出測試、執行、重跑失敗、解析報告、產生骨架、錄製操作。 AI 編輯器透過 MCP 協定自動呼叫。
RUNNERS
4 種測試框架
Playwright（Python）、Jest、Cypress（JS）、Go test。 以 QA_RUNNER 環境變數切換。
PLUGIN
可擴充架構
繼承 TestRunner 抽象類別、註冊至 registry， 即可新增 vitest、JUnit、xUnit 等框架支援。
REPORT
統一報告格式
各框架原生報告（pytest-json-report、Jest JSON、Mocha JSON、go test -json） 自動正規化為一致 schema 供 AI 解析。

MCP Tool 清單

Tool用途參數

技術架構

Plugin 化設計，每個測試框架自成一檔。

┌─────────────┐ ┌─────────────────┐ ┌──────────────┐ ┌──────────┐ │ AI 編輯器 │ → QA MCP Server │ → Runner │ → 測試框架 │ │ Claude │ (testbridge-runner) │ (plugin) │ pytest │ │ Cursor │ │ │ jest │ │ Windsurf │ │ │ ... │ └─────────────┘ └────────┬────────┘ └──────────────┘ └──────────┘ │ ↓ report.json → get_failure_details / get_test_report

目錄結構

testbridge-runner/
├── pyproject.toml
├── src/testbridge_runner/
│   ├── server.py            # MCP 入口（tool 註冊與路由）
│   ├── config.py            # 環境變數
│   ├── runners/             # Plugin 架構
│   │   ├── base.py          #   TestRunner 抽象介面
│   │   ├── pytest_playwright.py
│   │   ├── jest.py
│   │   ├── cypress.py
│   │   └── go_test.py
│   ├── tools/               # server → runner 薄層
│   └── resources/           # report:// / trace:// 資源
└── tests_project/           # 範例受測專案

新增 Runner 的步驟

  • 在 src/testbridge_runner/runners/ 新增模組
  • 繼承 TestRunner，實作 6 個 abstract method
  • 在 runners/__init__.py 的 REGISTRY 註冊一行
  • 更新 QA_RUNNER 環境變數即可生效

安裝與使用

三步驟接入 Claude Desktop。其他支援 MCP 的編輯器流程類似。

1. 安裝套件

# 取得原始碼
git clone https://github.com/your-org/testbridge-runner.git
cd testbridge-runner

# 建立虛擬環境
python3 -m venv .venv
source .venv/bin/activate

# 安裝依賴
pip install -e .

# 安裝 Playwright 瀏覽器（僅 pytest-playwright runner 需要）
playwright install

2. 設定 Claude Desktop

編輯 ~/Library/Application Support/Claude/claude_desktop_config.json：

{
  "mcpServers": {
    "testbridge-runner": {
      "command": "/絕對路徑/testbridge-runner/.venv/bin/python",
      "args": ["-m", "testbridge_runner.server"],
      "cwd": "/絕對路徑/testbridge-runner",
      "env": {
        "QA_RUNNER": "pytest",
        "QA_PROJECT_ROOT": "/絕對路徑/受測專案"
      }
    }
  }
}
NOTE路徑須為絕對路徑

command 必須指向虛擬環境內的 Python 執行檔； 若只填 "python"，Claude Desktop 會使用系統 Python，無法載入專案依賴。

3. 重啟並開始對話

重啟 Claude Desktop 後，左下角工具圖示應顯示 testbridge-runner 與 8 個可用工具。

# 範例對話
你：現在用哪個 runner？
你：列出所有測試
你：跑一下登入相關的測試
你：剛才哪幾個失敗？給我細節
你：幫我寫一個測試：訪問購物車頁，確認顯示 0 筆商品時的空狀態
你：重跑失敗的測試

切換測試框架

調整 QA_RUNNER 與 QA_PROJECT_ROOT 兩個環境變數即可。

# pytest + Playwright
QA_RUNNER=pytest
QA_PROJECT_ROOT=/path/to/python-project

# Jest
QA_RUNNER=jest
QA_PROJECT_ROOT=/path/to/node-project

# Cypress
QA_RUNNER=cypress
QA_PROJECT_ROOT=/path/to/cypress-project

# Go test
QA_RUNNER=go
QA_PROJECT_ROOT=/path/to/go-project

AI 編輯器看到的 tool 名稱與輸入介面不變， 底層執行的指令與產出的測試檔副檔名會自動切換。

與 TestBridge 雲端平台的關係

QA MCP Server 是 TestBridge 推出的本地端開源工具， 設計理念延續 TestBridge 主站的 BYOK（Bring Your Own Key）精神： 你的 AI 呼叫不經過我們的伺服器，測試資料也保留在你的開發機。

各自負責

  • QA MCP Server：本地端、單機、AI 編輯器對接、即時測試執行
  • TestBridge 雲端：團隊協作、測試案例管理、Bug 追蹤、儀表板、合規報告

未來規劃整合

  • 執行結果同步至 TestBridge 儀表板（規劃中）
  • 從 TestBridge 拉取雲端管理的 test case 至本地執行（規劃中）
  • 共用 19 個 AI Skill 作為 MCP Prompts（規劃中）
INDIE個人專案階段

TestBridge 與 QA MCP Server 目前皆為個人專案、公測階段，全功能免費。 如有想法或回饋，歡迎在 TestBridge 社群討論， 或直接來信。
