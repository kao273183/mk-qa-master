## §2 — Candidate themes (4)

### Theme B. v1.0 Stability Pass (promoted from v0.10 §2 / postmortem §9 recommendation)

> Stop iterating. Promise a stable contract.

What "stable" means:

- Lock the 21-tool surface — no name changes, no breaking arg removals
- Lock `report.json`, `plan_verification`, `Finding`, evidence-row shapes
- Document deprecation policy (one major version of warning before removal)
- Migration guide from 0.x → 1.0 (mostly: pin versions, watch for `keywords` changes)
- Stability test in CI (snapshot the schema; CI fails on diff without an explicit `BREAKING_CHANGE_ACK` env)

PR breakdown (3–4 PRs, ~5–6 days, +1 day vs v0.10-planning's estimate now that 5 tools are part of the bookend lock):

  - PR-1: Schema snapshot tests (tool descriptions, finding shapes, plan shape, evidence shapes per tool)
  - PR-2: Deprecation policy doc + migration guide
  - PR-3: README v1.0 section + bump to 1.0.0
  - PR-4 (optional): Deprecate any rough-edged 0.x APIs we want gone before v1

Risk: **low**. The bookend pattern from v0.10 is exactly the kind of shape that benefits from being explicitly frozen.

**Verdict (carries forward from v0.10 postmortem §9):** primary recommendation.

### Theme E. OWASP API4 (Rate Limit) — v0.8 deferred category, carried forward

> Close the gap from 5 of 10 to 6 of 10 OWASP API categories.

Unchanged from v0.10-planning §2 E:

- Low-amplitude burst pattern (≤10 req / 5 sec, then back off)
- Extra consent gate: `QA_API_SECURITY_RATE_LIMIT_CONSENT=true` (above the standard consent)
- Default OFF, opt-in per scan via `categories=[..., "rate_limit"]`
- Token-bucket aware: detect 429 / Retry-After and stop immediately

PR breakdown (2–3 PRs):

  - PR-1: `rate_limit` rule + responsible-probing test against the vulnerable fixture
  - PR-2: Stricter consent gate + docs
  - PR-3: README + version bump

Risk: **medium-high**. Most likely rule to be misused; consent gate must be ironclad.

**Verdict:** good v0.11 stretch goal IF Theme B has remaining budget. Otherwise hold for v0.12.

### Theme G. Edge AI Inference Runner (NEW — sourced from `mk-qa-master-edge-ai-enhancement.md`)

> Add an `edge` runner that drives RTSP-stream + inference-device test scenarios.

User-authored build-ready spec proposes an Edge AI runner — same pattern as the existing Maestro runner but for "RTSP source + inference device" instead of "Android device + adb host". Desktop mode (local YOLO) works without hardware; Jetson / remote-inference mode adds a network hop. New MCP tool `analyze_stream` mirrors `analyze_url` for video streams.

**Scope (build-ready spec, 4 phases):**

1. **Phase 1 — desktop YOLO**: `EdgeConfig` + RTSP source mgmt (mediamtx + ffmpeg) + `LocalYolo` backend + IoU/latency metrics + `EdgeInferenceRunner`. Working `QA_RUNNER=edge` pytest run against `fixtures/factory.mp4`.
2. **Phase 2 — analyze / generate**: `analyze_stream` MCP tool + Edge `generate_test` template + ground-truth sidecar loader.
3. **Phase 3 — remote inference**: `RemoteHTTP` backend + `QA_JETSON_HOST` health-check + Maestro-style timeout extension.
4. **Phase 4 — resilience + coach**: degradation injection (netem / 斷流 / 壞幀), Edge flake signals in `get_optimization_plan`, Edge knowledge section in `qa-knowledge.md`.

**New surface area**:

- 1 new runner (`edge`, aliased `rtsp`)
- 1 new MCP tool (`analyze_stream`) — total surface 21 → 22
- 1 new domain knowledge section
- 4 new modules under `src/mk_qa_master/edge/`
- New optional extras: `pip install "mk-qa-master[edge]"` (opencv-python, ultralytics, requests)

**Estimated effort (split across phases):**

| Phase | Days | Independently shippable? |
|---|---|---|
| 1 (desktop) | ~3d | Yes — `QA_RUNNER=edge` works on its own |
| 2 (analyze/generate) | ~2d | Yes — extends Phase 1 |
| 3 (remote) | ~2d | Yes — `QA_JETSON_HOST` switches backend |
| 4 (resilience) | ~2d | Yes — optimizer signals |
| **Total** | **~9d** | But ship-per-phase if scope tightens |

**Risk: medium-high.** Specifically:

1. **Heavy optional deps.** `ultralytics` pulls torch (~1GB), `opencv-python` is ~50MB. Optional extras mitigate but core users still discover the package.
2. **Hardware-specific paths inherently flaky.** Maestro runner taught us this — TCP timeouts, device-not-found, "QA_JETSON_HOST" reachability all add real CI burden.
3. **New CI matrix.** Phase 1's pytest run needs ffmpeg + mediamtx binaries + a baseline YOLO model file. Adds ~5min to CI per run; needs a path that skips when extras aren't installed.
4. **Surface bloat before v1.0 lock.** Adding `analyze_stream` (22nd tool) AND a new runner right before a stability pass means v1.0 promises a bigger surface than it would otherwise. Either accept the bigger lock OR sequence Edge AI as **v1.1+** explicitly.

**Strategic tension**: the spec's last paragraph is honest — "正是職務『從 0 到 1 建立 Edge AI 測試自動化框架』的活證據" (this is the live evidence of building an Edge AI testing framework from 0 to 1 for the job position). That's a meaningful personal-development goal that may or may not align with the user-base for whom v0.10 shipped. The themes B / C / E address **existing** mk-qa-master users; theme G opens a **new** user segment (edge inference QA).

**Verdict:** worth doing — the question is **when**. Two viable orderings:

- **G-then-B**: v0.11 = Edge AI (Phases 1-2 = MVP), v0.12 = v1.0 stability lock (now including the Edge surface). Means v1.0 ships with Edge AI promised.
- **B-then-G**: v0.11 = v1.0 stability lock with 21 tools, v0.12 = Edge AI as a 1.x extension. Means v1.0 ships sooner with a tighter surface.

## §3 — Recommendation

> **B-then-G**: ship v0.11 as the v1.0 stability lock with the current 21-tool surface, then ship v0.12 as Edge AI Phases 1-2 (or all 4).

Reasoning:

1. **v0.10 just landed — users need a stable target before another expansion.** v0.7 → v0.10 added 4 release lines (v0.7.x captcha, v0.8 API security, v0.9 plan/verify, v0.10 universal bookend). Every step was additive, but the cumulative drift is real. v1.0 with the current surface gives users (and ourselves) a known-good baseline.

2. **Edge AI is meaningfully different — sequence cleanly.** Theme G isn't a refinement of the current product; it's a new vertical. Bundling it into the v1.0 surface lock means a year from now we'd be back-revving the v1.0 spec because the Edge AI surface didn't quite fit. Easier to ship v1.0 with 21 tools + 7 runners + 4 hosts and add Edge AI as v1.1.

3. **Sequencing favors compound learning.** v1.0 stability pass forces us to write the schema-snapshot tests and migration guide that Edge AI will benefit from as it grows in v1.x. The reverse order (Edge AI first) means v1.0 has to retroactively retrofit those tests around the new surface — more work for the same final state.

4. **Carrier risk.** Edge AI's optional-extras chain (`ultralytics` → torch → ~1GB) is the kind of dep that can break PyPI uploads, Docker image sizes, or Glama plugin install. Best to introduce it as v1.x with the v1.0 stability mechanisms already in place to catch regressions.

5. **The career-portfolio motivation in the Edge AI spec is legitimate but separate.** The build-ready spec is excellent — when Edge AI ships, it will be high-quality. Holding it for v0.12 means it ships AFTER mk-qa-master has a v1.0 stability story; that's more credible than "edge AI feature on a pre-v1.0 product".

**Second-choice ordering**: if there's strong pull (e.g., a concrete job interview deadline) for Edge AI sooner, **G-then-B** is also viable — but commit to NOT mixing G into the v1.0 lock decision. Either G ships entirely pre-v1.0 (v0.11), OR entirely post-v1.0 (v0.12+). Don't half-ship G as part of v0.11 AND defer the rest to v1.x — that's the worst of both worlds.

## §4 — Anti-patterns (don't slip these into v0.11)

| Don't | Why |
|---|---|
| Lock v1.0 contract WHILE adding new runners | v1.0's value is "I can pin this version"; mixing surface changes into the lock PR undermines that |
| Defer schema-snapshot tests to v1.1 | The whole point of v1.0 is the snapshot; without it the version number is theater |
| Edge AI Phase 1 without Phase 2 | Phase 1 alone is "you can run YOLO via pytest" — no analyze_stream means host LLMs can't actually generate edge tests, so the user-facing story is incomplete |
| Skip the migration guide | v0.x → 1.0 with no migration doc means every user has to grep for what changed |
| Pin `ultralytics` / torch as required deps | Optional extras only; core install stays light |
| Remove env vars (only) — without YAML loader | Same as v0.10 §4 — strict-replacement breaks existing users. C is additive on purpose. |
