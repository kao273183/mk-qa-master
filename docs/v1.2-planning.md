## §2 — Candidate themes (5 active)

### Theme G Phase 3 — Remote Inference (primary recommendation)

> Complete Edge AI's "middle phase" by replacing the v1.1 `RemoteHTTP.infer()` stub with a real HTTP probe + Jetson health check.

The v1.1 code shipped with deliberate stubs:
- `RemoteHTTP.infer()` raises `NotImplementedError("v1.2 / Phase 3")` on call
- `EdgeInferenceRunner._healthcheck_device()` verifies URL *shape* but doesn't probe
- Users hitting either path see a clear error pointing at v1.2 — no silent failure

**Phase 3 scope** (per `mk-qa-master-edge-ai-enhancement.md` §4 + v1.1 PRD §3 deferred items):

- Wire `RemoteHTTP.infer(frame)`: encode frame via `cv2.imencode(".jpg", frame)`, POST as `multipart/form-data` to `cfg.inference_url`, parse JSON response into `InferResult(detections, latency_ms)`
- Real `GET /health` probe in `_healthcheck_device()` with `cfg.device_timeout_s` (default 60s; matches Maestro pattern for remote ADB)
- New error envelope on `inspect_stream` (no surface change) when the configured remote endpoint is unreachable at setup time — runner returns the existing `setup_failed` shape with a remote-specific hint
- Optional new env var: `QA_INFERENCE_TIMEOUT_S` for per-inference timeout (separate from `QA_DEVICE_TIMEOUT_S` which is setup-time)

**Surface implications under v1.0 lock**:
- Zero new MCP tools — runner internals only
- `QA_INFERENCE_TIMEOUT_S` would be a new env var → **minor bump v1.2.0** (additive)
- No changes to `analyze_stream` or `generate_test`
- Snapshot ack ONLY if a new env var lands (and even then, env vars don't appear in the tool surface snapshot — the doc-sync test catches the README env table)

PR breakdown (estimated):

| PR | Scope | Days |
|---|---|---|
| PR-1 | `RemoteHTTP.infer()` real implementation + 5 unit tests (mock `requests.post`) | 1 |
| PR-2 | `_healthcheck_device()` real probe + timeout handling + 4 tests | 0.5 |
| PR-3 | Optional `QA_INFERENCE_TIMEOUT_S` env var + EdgeConfig integration + 2 tests | 0.5 |
| PR-4 | Doc updates (README troubleshooting block now lists remote-mode errors; MIGRATION-1.x v1.1 → v1.2 entry) + version bump 1.1.2 → 1.2.0 + tag + release | 1 |
| **Total** | | **~3 days** |

**Risk: low.** v1.1 left clear stubs; v1.2 fills them in. The Maestro-style remote-target patterns are already in the codebase. Unit tests cover the `requests.post` mock cleanly.

**Verdict (carries forward from v1.1 postmortem §9):** primary recommendation.

### Theme G Phase 4 — Resilience + Coach (could bundle with Phase 3 OR split)

> Inject degradation scenarios + surface Edge-specific flake signals through `get_optimization_plan`.

Scope (from spec §11 + §4 of the enhancement doc):

- Degradation injection helpers: `netem` qdisc setup for network jitter / packet loss, mid-test ffmpeg kill for stream-disconnect, corrupted-GOP fixtures for bad-frame robustness
- Edge flake signals in optimizer's suite-quality lens: `latency_p95_exceeded_sla` (🔴 high), `fps_variance_across_runs` (🟡 medium), `iou_jitter_per_tc` (🟡 medium), `coverage_gap_per_label` (🟡 medium)
- New per-run fields in the report's edge metadata: histogram of latencies + per-frame IoU traces for failed tests

**Surface implications**:
- Zero new MCP tools
- New optional metadata field in `report.json` shape under an `edge_metrics` key → **additive minor bump** (snapshot ack DOES fire here because `get_test_report`'s response gains a documented optional field)
- New degradation helpers shipped as a Python module (`mk_qa_master.edge.resilience`) — not exposed as MCP tool; called by generated tests directly

PR breakdown:

| PR | Scope | Days |
|---|---|---|
| PR-1 | `mk_qa_master.edge.resilience` module: netem helper, kill_ffmpeg helper, corrupted_gop_fixture builder + 5 tests | 1.5 |
| PR-2 | Edge flake signals plumbed into `get_optimization_plan` + history archive includes `edge_metrics` + 4 tests | 1 |
| PR-3 | `generate_test` edge template adds optional resilience-mode flag + 2 tests | 0.5 |
| PR-4 | Doc updates + sample fixture corrupted-GOP variant + version bump + release | 1 |
| **Total** | | **~4 days** |

**Risk: medium.** netem requires root on Linux CI; macOS / Windows local dev can't use it. Need an OS-aware skip pattern. The corrupted-GOP fixture is small but needs careful ffmpeg invocation to produce something cv2 actually chokes on.

**Decision: bundle with Phase 3 as v1.2.0, OR split into v1.3.0?**

- **Bundle**: ~7 days, single release tells the Edge AI story end-to-end. Risk: bigger PR set, harder to bisect if something regresses.
- **Split**: v1.2.0 = Phase 3 (3 days, ships Real Soon Now); v1.3.0 = Phase 4 (4 days, post-v1.2 dogfood feedback). Risk: Edge AI story stays "in pieces" for an extra release cycle.

**Recommendation: split.** Phase 3's "remote inference works" milestone is independently shippable. Phase 4 benefits from one Phase-3 dogfood cycle to see what real failure modes surface.

## §3 — Recommendation: **Phase 3 as v1.2.0**

Reasons:

1. **Code stubs already in place.** v1.1's `RemoteHTTP` is import-safe and constructable; only `.infer()` raises. Filling it in is a constrained additive change with a clear contract (matches the `LocalYolo.infer()` return shape).

2. **The v1.1 release notes promised it.** Both `MIGRATION-1.x.md` and the v1.1.0 GitHub release notes explicitly call out "Phase 3 defers to v1.2.0+ — `NotImplementedError` raised if used early." Shipping v1.2.0 as Phase 3 delivers the promise.

3. **Smallest scope that ships a meaningful capability.** ~3 days, ~4 PRs, clear test plan, low risk. Compare to Phase 4 (~4 days, OS-aware CI work needed) or Theme C (~3 days, but harder to dogfood the backward-compat path).

4. **Sequences naturally before Phase 4.** Phase 4's resilience scenarios assume there's actually a remote target to be resilient against. Phase 3 first, Phase 4 against the Phase 3 dogfood findings.

5. **Earns one more `BREAKING_CHANGE_ACK` cycle for the stability mechanism.** v1.1.0 was the first ack-triggering release; v1.2.0 would be the second (the new env var is minor-bump additive, but the doc-sync test for README's env-vars table is the load-bearing check). Two exercises in three minor cycles is the cadence the mechanism was designed for.

**Secondary recommendation: bundle Theme C YAML config into v1.3.0 alongside Phase 4.** That way v1.x ends with: v1.2 = Phase 3, v1.3 = Phase 4 + YAML config, v1.4 = OWASP API4 (Theme E). Each release ships one user-facing change + one supporting infra change.

## §4 — Carry-forward housekeeping (load-bearing per §5 invariants)

Three items have now been flagged in MULTIPLE postmortems without landing. Per v1.1 postmortem §9 #2 ("two postmortems flagging the same gap = promote"), v1.2 must address them as part of its first PR:

1. **CI workflow check for `BREAKING_CHANGE_ACK=true` + migration-doc untouched.** Flagged in:
   - v0.11 postmortem §9 #1 (after v1.0 ship)
   - v1.0 postmortem §9 #1 (referenced forward)
   - v1.1 postmortem §9 #2 (explicit "promote to tracked")

   v1.2.0 PR-1 (or a dedicated tracking PR) adds a `.github/workflows/ci.yml` step that:
   - Detects `BREAKING_CHANGE_ACK=true` in the PR's CI env (set via workflow_dispatch or PR label)
   - Runs `git diff origin/main...HEAD -- 'docs/MIGRATION-*.md'`
   - Fails if the diff is empty AND the ack is set

   No more "next housekeeping" — lands in v1.2.0 or it's a v2.0 surface-lock regression risk.

2. **PR description templates.** Flagged in v0.10 postmortem §9 #4 + v0.11 postmortem §9 #2 + v1.1 (implicit reuse of v0.10's repeated PR-body structure). Single PR adds `.github/PULL_REQUEST_TEMPLATE/feat-bookend.md` + `feat-runner.md` + `feat-tool.md` + `release.md`. ~30 minutes. v1.2 housekeeping.

3. **`init_qa_knowledge` runner-aware Edge section selection.** v1.1 postmortem §9 #4. Currently the Edge Vision Inference section sits in `qa_context.py`'s universal methodology body; `init_qa_knowledge` doesn't know to surface it when `QA_RUNNER=edge`. Small additive change: starter template scaffolder reads `QA_RUNNER` and prepends a runner-specific section. **This IS a surface change** (return shape of `init_qa_knowledge` gains a `runner_section_included` field) → triggers snapshot ack. Bundle into v1.2.0's PR-3 or PR-4 to amortize the ack cycle.
