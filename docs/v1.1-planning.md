This doc opens after v1.0.0 shipped to PyPI on 2026-06-02. v1.0's stability lock means **every theme considered here lives under a stricter contract**: additive changes only, deprecation cycles for any rename or removal, snapshot tests catch drift.

## §2 — Candidate themes (3 active + 1 placeholder)

### Theme G. Edge AI Inference Runner (primary recommendation — promoted from v0.11 §2)

> Add an `edge` runner that drives RTSP-stream + inference-device test scenarios. Same pattern as the existing Maestro runner but for "RTSP source + inference device" instead of "Android device + adb host".

Build-ready spec exists at `mk-qa-master-edge-ai-enhancement.md`. 4 phases:

| Phase | Days | Independently shippable? |
|---|---|---|
| 1 (desktop YOLO) | ~3d | Yes — `QA_RUNNER=edge` works on its own |
| 2 (analyze_stream / generate) | ~2d | Yes — adds 1 MCP tool |
| 3 (remote inference) | ~2d | Yes — `QA_JETSON_HOST` switches backend |
| 4 (resilience + coach) | ~2d | Yes — optimizer signals |
| **Total** | **~9d** | |

**Constraint from v1.0**: each phase that touches the MCP surface (Phase 2 adds `analyze_stream`) is a **minor bump** under the v1.0 contract:
- `analyze_stream` is a new tool → tool count 21 → 22 (additive, minor-version OK)
- `BREAKING_CHANGE_ACK=true` required for snapshot rewrite
- README + SKILL.md + reference/tool-surface.md tool-count refs auto-flagged by v1.0 PR-2's sync test

So Theme G ships as v1.1.0 (Phase 1+2) → v1.2.0 (Phase 3) → v1.3.0 (Phase 4). OR all-phases-in-one as v1.1.0 if the dogfood holds. Either is valid; the v1.0 stability lock just makes the version progression mechanical.

**Risk: medium-high.** Unchanged from v0.11 §2 G:

1. Heavy optional deps (`ultralytics` → torch ~1GB)
2. Hardware-specific paths inherently flaky (TCP timeouts, device discovery)
3. New CI matrix (ffmpeg + mediamtx + baseline YOLO model)
4. Surface bloat now requires explicit minor bumps (v1.0 lock benefit: change is *visible*; downside: change *requires* version cycling)

**Verdict (carries forward from v0.11 §3):** primary recommendation. The build-ready spec, the carrier risk plan, and the career-portfolio narrative all still apply. v1.0's stability lock makes the rollout mechanical (no scrambling to fit it into v1.0's frozen surface).

## §3 — Recommendation: **Theme G as v1.1**

Reasons:

1. **B-then-G ratified in v0.11 §3.** The whole point of holding Theme G for post-v1.0 was so it lands as a v1.1+ minor under the stable contract, not a pre-v1.0 surface scramble. Backing out now means v0.11 §3's planning logic was wrong, which it wasn't.

2. **The spec is build-ready.** Unlike Themes C and E (which have open design questions on backward compat for YAML loader and consent gate design for rate-limit), Theme G's 4 phases each have concrete code, file paths, env vars, and acceptance criteria from `mk-qa-master-edge-ai-enhancement.md`.

3. **v1.0's stability lock makes G's progression visible.** Each phase that touches the MCP surface is a deliberate minor bump with snapshot test pairing. v0.x's "ship and see" loop becomes v1.x's "declare, snapshot, ship". Theme G is the first real test of that loop — better to find friction here than on a smaller theme.

4. **Career-portfolio narrative remains valid.** The "從 0 到 1 建立 Edge AI 測試自動化框架" framing from the spec's last paragraph is the kind of public artifact that benefits from shipping under a stable contract (`pip install "mk-qa-master[edge]"==1.1.0` is a sharper portfolio claim than `==0.11.0`).

**Sub-recommendation: ship Phases 1+2 together as v1.1.0.** Phase 1 alone is "you can run YOLO via pytest" — no MCP tool exposed yet. Phase 2 (`analyze_stream` tool) is what makes G feel like a v1.1.0 release rather than internal plumbing. Phases 3 and 4 land as v1.2.0 and v1.3.0 (or bundled — TBD when the v1.1 PRD opens).

## §4 — Anti-patterns (don't slip these into v1.1)

| Don't | Why |
|---|---|
| Skip the snapshot ack for Phase 2's `analyze_stream` addition | The whole v1.0 mechanism exists to flag this; bypassing it defeats the purpose |
| Pin `ultralytics` / torch as required deps | Optional extras only; v1.0 stability promise covers `pip install mk-qa-master==1.x` working on lightweight installs |
| Add `analyze_stream` AND another tool in the same minor | Each new tool is its own minor bump under v1.0 — bundling makes regression isolation harder |
| Remove the `edge` runner mid-v1.x because someone reports it's slow | Removals need the deprecation cycle (≥ 1 minor of warning); cf. DEPRECATION-POLICY.md |
| Drop the snapshot test in v1.0 to ease G's rollout | Cf. the v1.0 stability promise. The friction IS the feature. |

## §9 — Postmortem: Theme G Phases 1+2 shipped as v1.1.0 + v1.1.1 (2026-06-03)

**Status:** Closed. v1.1.0 shipped to PyPI on 2026-06-03 (Edge AI Runner functional surface); v1.1.1 followed the same day with the housekeeping items the v1.1.0 PR-4 explicitly deferred (sample fixture + CI job + Edge knowledge section). Both via the standard `publish.yml` OIDC flow. PRD: `docs/prd-v1.1-edge-ai-runner.md`.

### What landed

| Release | PRs | Scope | Tests | Notes |
|---|---|---|---|---|
| v1.1.0 | #81 PR-1 | `EdgeConfig` + `edge/` module tree + `[edge]` extras | +40 | Heavy deps (`ultralytics` → torch) mocked via `patch.dict("sys.modules", ...)` so tests run on base install |
| v1.1.0 | #82 PR-2 | `EdgeInferenceRunner` + REGISTRY aliases (`edge`, `rtsp`) | +18 | Caught + fixed a same-PR bug: env build was happening before `setup()` ran |
| v1.1.0 | #83 PR-3 | `analyze_stream` MCP tool + spec §10 edge template + vendor-host blacklist | +12 net | **First real exercise of v1.0's `BREAKING_CHANGE_ACK` mechanism** — gate worked |
| v1.1.0 | #84 PR-4 | `docs/MIGRATION-1.x.md` + README v1.1 section + version bump 1.0.0 → 1.1.0 | n/a | 22nd tool registered; 4 manifests synced |
| v1.1.1 | #85 | Sample fixture (75KB ffmpeg testsrc) + `edge-sample` CI job + EN/zh-TW Edge knowledge section | n/a | Patch-only — zero MCP surface change; honored DEPRECATION-POLICY strictly |

5 PRs across 2 releases. v1.1.0 ~6 working days (matched PRD §8's 5-day estimate plus a half-day for the stability-lock paperwork that v0.x didn't need). v1.1.1 ~1 day. 389/389 tests pass on PyPI v1.1.1.

### What worked

1. **The v1.0 stability lock paid off on first real use.** PR-3 was the first release to trigger `BREAKING_CHANGE_ACK=true` against a genuine surface change. The mechanism behaved exactly as designed: drift detection refused without the ack, refused again with the ack but without paired docs, then rewrote the snapshot when ack + docs were all present. Doc-sync test caught every stale "21 tools" reference and required a 21 → 22 sweep through README + SKILL.md + reference/*.md. **The v1.0 work in PR-1..3 of that release wasn't theater — it earned its keep within 24 hours.**

2. **Bundled spec → mini-PRD → ship loop scales.** `mk-qa-master-edge-ai-enhancement.md` arrived as a complete build-ready document. The mini-PRD at `docs/prd-v1.1-edge-ai-runner.md` consumed it as the technical reference and added only: scope demarcation (which phases ship), v1.0 stability lock paperwork callouts, decision ratifications. The PRs then mechanically implemented against the spec without ad-hoc scope creep. Same-day debugging caught one bug (PR-2's env build order); the rest of the spec implemented exactly as written.

3. **Patch-vs-minor discipline held under pressure.** v1.1.0's PR-4 deferred 3 items to "v1.1.1 housekeeping" with explicit reasoning in the commit + release notes (binary asset + heavier CI infra). v1.1.1 then landed all three as a clean patch with zero surface change. The DEPRECATION-POLICY.md table ("what counts as breaking") was the load-bearing reference — every contributor decision checked against it.

4. **Heavy-dep test isolation pattern proved out.** `patch.dict("sys.modules", {"ultralytics": MagicMock(YOLO=fake_yolo)})` let PR-1's 40 unit tests run on a base install. PR-3's `analyze_stream` tests used the same pattern for `cv2`. The CI job in PR-4 (v1.1.1) installs cv2 but skips ultralytics + torch, relying on the mocked-unit-test coverage for the YOLO path. Saved roughly 1 GB of CI install per job + 5+ minutes of wall-clock.

5. **Vendor-host blacklist landed before anyone could misuse it.** PRD §11 #6 chose "ship in v1.1" over "hold until reported abuse". The blacklist (Dahua / Hikvision / Ezviz / Axis / Amcrest / Lorex / Swann / Reolink) + `QA_EDGE_ALLOW_VENDOR_HOSTS` override are in place from day one. Better than the v0.7 path where consent gates evolved release-by-release as new abuse vectors surfaced.

6. **`edge-sample` CI job went green on first try (59s).** Apt-install ffmpeg + pip-install cv2 + ffprobe sanity + 7 edge test files + analyze_stream plumbing dogfood — all wired correctly without iteration. The synthetic fixture (75KB testsrc clip + hand-written annotations) is small enough to commit and structured enough to exercise the entire `analyze_stream → labels[]` path.

### What we'd do differently

1. **The PR-2 env-build-order bug was avoidable.** PR-2's initial `run_tests()` built the `EDGE_*` env dict BEFORE `setup()` ran, so `EDGE_RTSP_URL` got the raw file path instead of the local RTSP URL. Caught by a single test on first pytest run, but the bug shouldn't have existed — the spec §7 fixture imports clearly assume the env points at an active RTSP URL. **Lesson:** When porting setup/teardown lifecycle, write a single integration assertion against the post-setup state BEFORE the implementation. The fix took 30 seconds; designing the test that caught it would have taken 30 seconds upfront too — but the bug-fix commit becomes "implementation only" instead of "implementation + remediation".

2. **The CI workflow ack-check from v0.11 postmortem §9 #1 STILL hasn't landed.** v0.11 postmortem flagged a `.github/workflows/ci.yml` step that fails when `BREAKING_CHANGE_ACK=true` is set but `git diff origin/main...HEAD docs/MIGRATION-*.md` is empty. v1.0 postmortem §9 #1 carried it forward as v1.1+ work. v1.1 PR-3 had the perfect opportunity to add it (it was the first ack-triggering release). Skipped because it wasn't blocking. **Lesson:** "Defer to next housekeeping" only stays honest if there's a next housekeeping. Two postmortems flagging the same gap is the signal that "next housekeeping" isn't materializing — promote to a tracked item before v1.2 planning closes.

3. **README "Edge AI Runner (v1.1.0)" section is short by design — but Glama will probably reduce its score.** v1.0 postmortem §9 #3 noted the Stability Promise section was wordy; v1.1.0 over-corrected with a ~30-line section. Glama's heuristics favor longer, structured READMEs with examples. **Lesson:** README sections want a sharp 1-paragraph hook BUT also a complete end-to-end example + a troubleshooting block. v1.1.x housekeeping should expand the Edge AI section with: (a) a complete `analyze_stream → generate_test → run_tests` walkthrough; (b) a `troubleshooting` block for the common mediamtx / ffmpeg / RTSP "open failed" classes of error.

4. **`init_qa_knowledge` doesn't auto-load the Edge section in v1.1.1.** The new EN + zh-TW Edge knowledge content sits in `qa_context.py`'s `_UNIVERSAL_METHODOLOGY_*` bodies. When `QA_RUNNER=edge` is active, the spec §13 promised init_qa_knowledge would specifically surface the Edge section in the scaffolded `qa-knowledge.md`. v1.1.1's section addition makes the content available via `get_qa_context(section="Edge Vision Inference")`, but `init_qa_knowledge` doesn't auto-select it based on QA_RUNNER. **Lesson:** This is a small additive change (runner-aware section selection in the starter template), but it's a real surface change → minor bump. Track for v1.2.

5. **`generate_test` template's `EDGE_TEST_TEMPLATE` is hardcoded as a constant.** Easier to maintain inline for now, but other runners (Maestro, Cypress) keep their templates in dedicated `templates/` files. **Lesson:** Migrating EDGE_TEST_TEMPLATE to `src/mk_qa_master/templates/edge/test_*.py.tmpl` would let it be lint-checked as actual Python. Defer to v1.2 if Phase 4's resilience scenarios add more variants.

6. **The vendor-host blacklist is static — no telemetry on how often it fires.** Privacy-wise the right default (telemetry would log RTSP URLs, which is exactly the surveillance footage we're trying to keep out of telemetry), but operationally it means we don't know whether the list is too aggressive or too permissive. **Lesson:** Document the override pattern more prominently and trust the v1.0 stability cycle — additions to the blacklist are minor bumps, well-publicized.
