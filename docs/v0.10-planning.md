# mk-qa-master v0.10 — Planning Notes

**Status:** Decision-support doc, NOT a locked PRD · **Author:** Jack Kao · **Date filed:** 2026-05-27 · **Successor to:** v0.9.x bookend series (v0.9.0 #64 → v0.9.5 #69)

This doc captures the v0.10 candidate themes after v0.9.x shipped to PyPI. It is **planning material**, not a contract — themes are picked, not promised. Once one is selected, that theme gets its own mini-PRD per the discipline established in [`docs/prd-v0.8-api-security.md`](prd-v0.8-api-security.md).

## §2 — Candidate themes (5)

### Theme A. Universal Bookend

> Make `plan_id` a first-class arg on every meaningful tool, not just the API security scanner.

The v0.9.x investment becomes "the way you use mk-qa-master" instead of "a special API-security thing."

Tools to extend:

| Tool | New plan-verification meaning |
|---|---|
| `run_tests(plan_id=…)` | CPs declare "test X passes / nodeid Y fails / suite duration < N seconds"; auto-verify after the pytest run |
| `solve_visual_challenge(plan_id=…)` | CPs declare "status='passed' AND token populated"; verifies CAPTCHA actually solved |
| `analyze_url(plan_id=…)` | CPs declare expected modules (form / cta / tab_bar) discovered |
| `auto_generate_tests(plan_id=…)` | CPs declare "≥ N test files generated, each covers a module from analyze_url" |

PR breakdown (5 PRs, ~6–8 days):

  - PR-1: `run_tests(plan_id=…)` + dogfood with pytest report.json as the evidence stream
  - PR-2: `solve_visual_challenge(plan_id=…)` + dogfood against the v0.7 fixture
  - PR-3: `analyze_url(plan_id=…)` + dogfood against a known-shape page (or the sample_api_project)
  - PR-4: `auto_generate_tests(plan_id=…)` + dogfood
  - PR-5: v0.10.0 release — README update, SKILL.md narrative, version bump

Risk: low — every PR is a mechanical copy of the v0.9.4 pattern. Bigger risk is design drift if we don't standardize how each tool exposes its findings as evidence rows.

### Theme B. v1.0 Stability Pass

> Stop iterating. Promise a stable contract.

What "stable" means:

- Lock the 21-tool surface — no name changes, no breaking arg removals
- Lock `report.json`, `plan_verification`, `Finding` schemas
- Document deprecation policy (one major version of warning before removal)
- Migration guide from 0.x → 1.0 (mostly: pin versions, watch for `keywords` changes)
- Stability test in CI (snapshot the schema; CI fails on diff without an explicit `BREAKING_CHANGE_ACK` env)

PR breakdown (3–4 PRs, ~4–5 days):

  - PR-1: Schema snapshot tests (tool descriptions, finding shapes, plan shape)
  - PR-2: Deprecation policy doc + migration guide
  - PR-3: README v1.0 section + bump to 1.0.0
  - PR-4 (optional): Deprecate any rough-edged 0.x APIs we want gone before v1

Risk: locking too early. The bookend pattern from A is the kind of thing that should belong in the v1.0 surface — doing B before A means we'd ship v1.0 with `plan_id` only on one tool, then have to break compat later to add it everywhere.

**Verdict:** B should follow A, not precede.

### Theme C. YAML Config UX

> Stop the env-var sprawl.

Current env-var count:

```
QA_RUNNER QA_PROJECT_ROOT QA_LANG QA_ANDROID_HOST QA_TIMEOUT_SECONDS
QA_VISUAL_CHALLENGE_CONSENT QA_VISUAL_CHALLENGE_AUTHORIZED_DOMAINS
  QA_VISUAL_CHALLENGE_TIMEOUT
QA_API_SECURITY_CONSENT QA_API_SECURITY_AUTHORIZED_DOMAINS
QA_PLAN_PERSIST MK_QA_PLANS_DIR MK_QA_REPORT_PATH
```

13 env vars. Hard to remember, hard to version-control, hard to share across team. Single YAML file fixes all three:

```yaml
# mk-qa-master.yaml
runner: pytest
project_root: ~/my-test-project
timeout_seconds: 600
language: en
android_host: 127.0.0.1:5555

visual_challenge:
  consent: true
  authorized_domains: [staging.example.com]
  timeout: 120

api_security:
  consent: true
  authorized_domains: [api.example.com]
  default_categories: [headers, broken_auth, bola, function_authz]

plan:
  persist: true
  plans_dir: ./test-results/plans
```

`QA_CONFIG_FILE=mk-qa-master.yaml` activates the loader. Existing env vars stay as explicit overrides (highest precedence).

PR breakdown (2–3 PRs):

  - PR-1: Config loader (precedence: env var > YAML > built-in defaults)
  - PR-2: `init_qa_knowledge`-style scaffolder that writes a starter `mk-qa-master.yaml` from current env state
  - PR-3: SKILL.md + README updates + v0.10.0 release

Risk: medium. Backward compat needs care — current users have env vars baked into their MCP wiring. Plan: env vars keep working forever (highest precedence), YAML is purely additive.

### Theme D. Cloudflare Turnstile (third CAPTCHA vendor)

> Cover the post-reCAPTCHA / post-hCaptcha era.

Turnstile is what's replacing reCAPTCHA on a lot of sites. Two interaction modes:

- **Passive (80%+ of cases):** server-side scoring, no visible challenge. Token granted invisibly.
- **Interactive (~20%):** small widget asks for a single click or shows an image challenge.

Implementation maps cleanly to the existing fingerprint table:

```python
_FINGERPRINTS.append({
    "id": "turnstile",
    "iframe_selectors": [
        'iframe[title*="Cloudflare"]',
        'iframe[src*="challenges.cloudflare.com"]',
    ],
    "verify_button_selector": ".cf-turnstile-checkbox",
    "response_token_selector": 'input[name="cf-turnstile-response"]',
    ...
})
```

PR breakdown (4–5 PRs):

  - PR-1: Fingerprint addition + passive-token detection (just wait for the textarea to fill)
  - PR-2: Interactive widget — single-click flow
  - PR-3: Image-challenge variant — reuse the reCAPTCHA tile-grid path
  - PR-4: Dogfood against a public Turnstile demo page
  - PR-5: v0.10.0 release

Risk: high. Two factors:

1. ~80% of Turnstile traffic is passive — AI vision adds no value. The MCP surface starts looking misaligned ("solve" semantically wrong when there's nothing to solve).
2. Cloudflare actively fingerprints Playwright. Our visual challenge already gets caught on real reCAPTCHA (see v0.7 success-rate caveat in README). Turnstile is harder.

**Verdict:** D is the kind of "sounds modern" addition that ships fragile. Pass for v0.10 unless a real user is blocked.

### Theme E. OWASP API4 (Rate Limit) — v0.8 deferred category

> Close the gap from 5 of 10 to 6 of 10 OWASP API categories.

v0.8 PRD §3 deferred API4 because rate-limit probing risks DoS. v0.10 can revisit if we design "responsible probing":

- Low-amplitude burst pattern (≤10 req / 5 sec, then back off)
- Extra consent gate: `QA_API_SECURITY_RATE_LIMIT_CONSENT=true` (above and beyond the standard consent)
- Default OFF, opt-in per scan via `categories=[..., "rate_limit"]`
- Token-bucket aware: detect 429 / Retry-After headers and stop immediately

PR breakdown (2–3 PRs):

  - PR-1: `rate_limit` rule + responsible-probing test against the vulnerable fixture (add a rate-limited endpoint with a low ceiling so we don't accidentally test "no limit at all")
  - PR-2: Stricter consent gate + docs
  - PR-3: README + version bump

Risk: medium-high. Even with all the safeguards, this is the rule most likely to be misused. The consent gate has to be ironclad.

## §3 — Recommendation: **Theme A (Universal Bookend)**

Reasons:

1. **Lowest cognitive switching cost.** Host LLM already learned `qa_plan → tool(plan_id=…)`; extending it to 4 more tools is mechanical.
2. **Every PR ships value alone.** Unlike B (v1.0 stability — must be cohesive) or C (YAML config — must replace all env vars to be useful), A's PRs are independently merge-mergeable.
3. **Pays back the v0.9.x investment.** qa_plan currently covers 1 of 21 tools (5%). After A, it covers 5 of 21 (24%). Big jump in everyday utility.
4. **Naturally precedes B.** A locks the bookend semantics across the surface; B can then promise "this is what v1.0 looks like" with confidence.

Second-choice: **Theme C (YAML Config)** if A's scope feels too big.

## §4 — Anti-patterns (don't slip these into v0.10)

| Don't | Why |
|---|---|
| GraphQL support | No user demand; "sounds modern" trap. |
| GUI / dashboard | `report://html` covers the use case. GUI is v2.0+ work. |
| AI-driven test generation (deeper than `auto_generate_tests`) | Host LLM already does this; runner shouldn't take over. |
| Removing env vars (only) — without YAML loader | Strict-replacement breaks existing users. C is additive on purpose. |
| Adding more `_skip_when_*` flags to existing tools | Surface grew enough in v0.9; consolidate before expanding. |

## §5 — Process invariants to carry forward (lessons accumulated v0.7 → v0.9)

1. **Mini-PRD before action**, period. Every theme picked here gets its own `docs/prd-v0.10-<theme>.md` before PR-1. Same shape as `prd-v0.8-api-security.md` (12 sections, open questions explicit).
2. **Dogfood against real artifacts.** Mocks catch orchestration; real fixtures catch bad assumptions. The v0.8 mobile postmortem said it; every v0.9.x feature observed it.
3. **Version-sync invariant earns its keep.** `test_plugin_versions_match_pyproject` fired on every v0.9.x PR. Don't remove. Don't loosen.
4. **PyPI Summary 512-char limit** is now tested. Don't reintroduce the regression (see v0.9.5 hotfix).
5. **Spike validation must assert produced VALUE, not just rc=0.** v0.8 mobile lesson, still load-bearing.
6. **Every consent gate documents itself.** `consent_required` envelopes echo the legal disclaimer verbatim — host LLM is required to surface them, not paraphrase.
