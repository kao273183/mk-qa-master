### Theme G Phase 4 ‚Äî Resilience + Coach (primary recommendation, carried from v1.2)

> Complete Edge AI's "story arc" by shipping degradation injection + Edge flake signals into `get_optimization_plan`.

Scope (unchanged from v1.2 ¬ß2 G Phase 4):

- Degradation injection helpers: `mk_qa_master.edge.resilience` module
  - `netem` qdisc setup for network jitter / packet loss (Linux only ‚Äî covered by OS-aware skip)
  - Mid-test ffmpeg kill helper for stream-disconnect scenarios
  - Corrupted-GOP fixture builder (small ffmpeg invocation to produce a clip cv2 will choke on partway through)
- Edge flake signals plumbed into `get_optimization_plan`'s suite-quality lens:
  - `latency_p95_exceeded_sla` (üî¥ high)
  - `fps_variance_across_runs` (üü° medium)
  - `iou_jitter_per_tc` (üü° medium)
  - `coverage_gap_per_label` (üü° medium)
- New `edge_metrics` key in `report.json` (additive optional field ‚Äî triggers `BREAKING_CHANGE_ACK=true` because `get_test_report`'s response shape gains a documented optional field)
- `generate_test` edge template extension: optional resilience-mode flag that injects a `netem`-triggering subprocess call at session-setup

**Surface implications**:
- Zero new MCP tools
- One documented response-shape addition (`edge_metrics` in `get_test_report` response) ‚Üí snapshot ack
- New helper module (`mk_qa_master.edge.resilience`) ‚Äî not exposed as MCP tool; called by generated tests directly
- One new optional env var: `QA_EDGE_NETEM_ENABLED` (default false; only Linux supports it)

PR breakdown:

| PR | Scope | Days |
|---|---|---|
| PR-1 | `mk_qa_master.edge.resilience` module: 3 helpers + 5 unit tests (subprocess mocked) | 1.5 |
| PR-2 | Edge flake signals plumbed into `get_optimization_plan` + history archive `edge_metrics` field + 4 tests | 1 |
| PR-3 | `generate_test` edge template resilience-mode flag + 2 tests | 0.5 |
| PR-4 | OS-aware CI: `edge-sample` job gains a `netem`-conditional step (skipped on macOS); README + MIGRATION-1.x entry + version bump + release | 1 |
| **Total** | | **~4 days** |

**Risk: medium.** Specifically:
1. `netem` requires root on Linux CI; the OS-aware skip pattern needs to gate both the helper's tests AND the CI runner step
2. Corrupted-GOP fixture is small but needs careful ffmpeg invocation to produce something cv2 actually chokes on
3. New `edge_metrics` response-shape change triggers snapshot ack (3rd time the mechanism fires ‚Äî proves it scales)

**Verdict: primary recommendation for v1.3.** Closes the Edge AI Phase arc, lets v2.0 ship with a complete Edge AI story.

## ¬ß3 ‚Äî Recommendation: **Phase 4 as v1.3.0**

Reasons:

1. **Closes the Edge AI Phase arc cleanly.** Phases 1+2 (v1.1), 3 (v1.2), and 4 (v1.3) makes v1.3 the natural Edge AI completion point. Anyone landing on v1.3 sees the full feature surface.

2. **Right time to fire the snapshot ack mechanism a third time.** `edge_metrics` response field addition triggers ack. v1.0's stability mechanism has now been exercised twice (v1.1.0 + reasoning at v1.2.0's `init_qa_knowledge` response addition which didn't fire because snapshot is inputSchema-only). v1.3.0 firing it a third time confirms the pattern scales.

3. **Spec is build-ready.** `mk-qa-master-edge-ai-enhancement.md` ¬ß11 + ¬ß12 still apply. The optimizer integration follows the v0.5 self-improvement-coach surface that's been stable since v0.5.

4. **4 working days is the right v1.3 scope.** Bigger than v1.2.1 (docs only), smaller than v1.1.0 (Edge AI Phases 1+2 took 7 working days). Lets v1.4 be Theme E (rate-limit) before v2.0 work opens.

5. **No competing themes with similar urgency.** Theme C is a "we'd be tidier" not "users are asking". Theme E benefits from v1.4 standalone. Theme I doesn't exist yet. Theme J is sweepable into the release PR.

**Sub-recommendation: bundle Theme J's cheap items into v1.3.0's PR-4 (release PR).** Specifically the tool-surface.md "stable since v1.x" addition for `analyze_stream`. ~5 minutes; honors the spirit of "release PR sweeps stale doc claims".

## ¬ß4 ‚Äî Anti-patterns (don't slip these into v1.3)

| Don't | Why |
|---|---|
| Bundle Phase 4 + Theme C YAML config in v1.3 | Two surface-change PRs in one release ‚Üí review burden + harder bisect if something regresses |
| Ship Phase 4 without `edge_metrics` snapshot ack discipline | The ack mechanism exists for exactly this ‚Äî flag the change, document the migration, prove the cycle |
| Add a new MCP tool in v1.3 | Tool count stays at 22; v1.3 is closing the Edge AI arc, not opening a new one. New tools wait for v2.0 |
| Make `netem` mandatory (Linux-only blocking) | OS-aware skip is the right pattern; users on macOS/Windows shouldn't need a Linux VM for Phase 4 |
| Relicense to Apache 2.0 in v1.3 | The deprecation cycle says ‚â• 1 minor of announcement first. v1.2.1 announced; v1.3 holds; v2.0 relicenses. Skipping the hold violates the policy and undermines the mechanism it was supposed to prove |
| Skip v1.x bugfix maintenance after v2.0 ships | DEPRECATION-POLICY says ‚â• 6 months of v1.x.y patches under MIT after v2.0 lands. Plan accordingly when scoping v2.x and v1.x branches |
